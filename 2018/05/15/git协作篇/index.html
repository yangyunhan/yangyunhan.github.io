git协作篇

发表于 2018-05-15 | 分类于 skill |
创建一个新项目分支

假设新建项目分支名为：projectName，执行步骤：

  • 切换到本地主干分支：git checkout master
  • 拉取主干最新代码： git pull
  • 从最新主干切出项目分支： git checkout -b projectName
  • 推送到远端 git push -u origin projectName (加上-u 参数表示关联远端，远端分支名和本地保持一致)
项目迭代

如果projectName这个项目开发完成上线，现需要二期开发，直接从projectName切出一个二期分支projectName-xx

二期新功能在projectName-xx分支上开发，开发完成合并到projectName，然后上线。

从远端切出一个分支

如果远端已经存在一个分支projectName-test，此时你需要从远端拉取这个分支进行开发，操作如下：git checkout --track origin/projectName-test 。该操作的意思是：直接从远端拉取这个分支到本地，新建一个分支projectName-test，并且和远端保持关联。

本地分支和远端分支保持关联的作用：

如果本地分支关联了远端分支，本地分支推送到远端的时候可以不用指定远端分支名，直接git push

如果本地分支没有关联远端分支，本地分支推送到远端的时候需要手动指定远端分支名，例如： git push origin xxxName

如何关联远端分支

如果在本地新建一个分支名为project-name的分支，首次推送到远端的时候加上 -u 参数，即可和远端建立关联。例如：git push -u origin project-name，下次推送改分支的时候直接git push即可。

在从远端切出一个分支的时候可以直接建立关联，使用git checkout --track origin/projectName-test ，本地projectName-test和远端projectName-test就直接建立了关联。

常用命令：
  • git branch -a 查看所有分支
  • git checkout <分支名> 切换到对应分支上
  • git checkout -b <分支名> 新拉一条<分支名>的分支
  • git reset –hard 【版本号前4位】回退到对应的版本
  • git reset –hard HEAD^ 回退到上一个版本，一个^ 表示一个版本，如果为^^^ 则表示，回退到上上上个版本，以此类推
git 提交

原则：少量多次提交。

提交的时候，标明commit的类型，要保持良好的规范，包括以下几种：

  • feat：新功能
  • fix：修复问题
  • docs：修改文档
  • style：修改代码格式，不影响代码逻辑
  • refactor：重构代码，理论上不影响现有功能
  • perf：提升性能
  • revert：回退，建议直接使用Github Desktop回退，而不是使用命令
  • logic：逻辑完善

git冲突解决

1）使用git pull文件时与本地文件冲突：

  • git stash 将本地修改存储起来
  • git stash list可以看到保存的信息 stash{0}就是刚才保存的标记
  • git stash pop stash@{0} 还原暂存的内容

首先git pull发现冲突，这时候记录下提交的commit id，然后git reset --hard到一个较早的历史版本，git pull到最新代码。

2）分支合并产生冲突：

场景：新建分支修改提交，切换到master分支，在master分支上修改并提交。此时git无法执行快速合并，会把各自的修改合并起来，合并起来可能有冲突，需要手动解决。

问题：

error: Your local changes to the following files would be overwritten by merge: protected/config/main.php

Please, commit your changes or stash them before you can merge.

如果希望用代码库中的文件完全覆盖本地工作版本。方法如下:

  • git reset –hard
  • git pull

3）撤销一个合并：git merge –abort

4）撤销合并后出现一个错误，想回到合并之前的版本：git reset –hard
