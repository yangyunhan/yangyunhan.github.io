初识PWA新风潮

发表于 2018-10-27 |

PWA

Progressive Web App(PWA) 上海 2017.11 google

渐进式网页应用，包含技术：App Manifest、Service Worker、Web Push等。

优势：融合了Web和App各自的优势，如渐进式（适用于选用任何浏览器的所有用户）、自适应（适合任何机型）、可离线（离线或低质量网络）、实现类似App的交互、及时更新、安全（https）、可以被搜索引擎检索、可安装（不必使用应用商店）、可链接。

其中关键的技术：

（1）可安装：通过提供manifest.json文件，并在html页面以<link rel="manifest" href="/manifest.json">的方式引用。

实现效果：在主屏幕上添加图标，点击启动。

（2）离线使用：Service worker

角色是客户端代理，是一个独立的线程，不参与web交互行为。是事件驱动的worker，生命周期与页面无关。（其他两种Dedicated Worker和Shared Worker关联的页面关闭，生命周期就结束）

注册一个Service worker

Service worker有生命周期：

Parsed – Installing – Installed – Activating – Activated – Redundant

​ 离线缓存 移除旧的缓存 处理功能函数 被另一个代替

实现效果：离线使用、消息推送

怎么知道网页是否具有PWA的特点：Chrome浏览器开发者工具中的Audits面板，帮助检测PWA的特性。

已经使用的网站：豆瓣、饿了么

（3）消息推送

在HTTP2中：

推送什么：推送静态资源不是一件有意义的事，因为html文件不大，请求的实际数据很小，请求时间可以忽略不计，耗费自己服务器的带宽，被缓存的资源大多数情况下反而会浪费流量。所以推送的资源是API，浏览器更难发现API，一般不缓存的特点。服务器对推送的资源有限制：请求是可缓存的、安全的，不能带请求体。所以server push可以推送GET和HEAD请求的响应。

怎么推：服务器构造请求，填充到PUSH_PROMISE帧里发送给客户端，告知客户端已经代劳发了这个请求，客户端看到这个帧，即使服务器还没推完，也不会发出这个请求。但是PUSH_PROMISE在html之前。

在PWA中：

推送和通知使用的是两个API：Notification API，负责所有与通知本身相关的机制；Push API定义了web应用如何向推送服务发起订阅、如何响应推送消息，以及web应用、应用服务器与推送服务器之间的鉴权与加密机制；由于Push API并不依赖 web 应用与浏览器 UI 存活，所以即使是在 web 应用与浏览器未被用户打开的时候，也可以通过后台进程接受推送消息并调用 Notification API 向用户发出通知。但是目前的支持情况还不好。

（4）缓存管理

Service worker Cache API将每次从服务端取回的数据在service worker处做了一次缓存。下次客户端再获取相同的请求或资源或在离线情况下请求资源时，可以直接从service worker中读取。

（5）网络请求

fetch前，js通过XHR发起网络请求，但是xhr有局限性，不支持Request、Response对象，也不支持streaming response，受跨域资源限制。Service worker通过拦截fetch请求，判断是从缓存读取数据还是将请求发往服务器。

PRPL

Google工程师提出一种web应用架构模式：PRPL(Push/Preload、Render、Precache、Lazy-Load)。

Push/Preload：推送/预加载初始URL路由所需的关键资源，HTTP2的Server Push或<link rel="preload">

Render：渲染初始路由，尽快让应用可被交互

Precache：预缓存路由，实现离线加载

Lazy-Load：按需加载、懒实例化
