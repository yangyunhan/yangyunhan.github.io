带着Webpack走上自动化之路

发表于 2018-03-28 | 分类于 skill |

定位

  Webpack是前端资源加载/打包工具，模块化解决方案。

  其他的关于前端构建工具还有：

  • 编译器：Babel

  • 任务运行器：Gulp/Grunt，定义并运行任务的工具，编写模块打包器不能处理的任务是有必要的（为模块打包器自动生成的静态资源文件在网页中插入一个script标签）。Grunt是先前定义过的任务的工具，通过一系列的配置对象声明式定义。缺点是对插件运行效果做微调的话，通过编写JavaScript代码没法实现，需要重写一个Grunt插件。Gulp使用JavaScript的函数定义任务，知道如何处理JavaScript返回的streams和promises。缺点是streams和promises的对接较难理解。

  • 模块打包器：Browserify等。Browserify使得Node packages获得浏览器支持，也可以帮我们把Web应用打包成一个Node Packages。但是单页应用通常需要的资源如CSS、图片及字体是Node不需要的，尽管有插件可以帮助解决这个问题，但是这些插件违背了该工具的设计初衷，配置将会比较混乱。

适用场景

  • 处理代码依赖。当项目复杂，需要引入许多静态资源，而且静态资源有依赖关系，优先关系时；
  • 代码的并包（多个静态资源合并，在进行模块化开发一定会用到）；
  • 代码的拆包（当你的资源需要多个页面复用而需要把共同的代码分离，以便用于浏览器缓存时）；
  • 代码构建，需要区分不同的生成环境时。（比如最常见的三层环境。开发环境、测试环境、线上环境。你可以在开发环境搭建hmr来提高编译效率，在测试环境使用soucemap来方便测试debug，在线上环境压缩混淆版本管理来减小带宽消耗和利用缓存）；

优势

  • 统一了JS的各种模块系统，取代了Browserify、RequireJS、SeaJS的工作.

  • 可以生成Source Maps（使调试更容易）：帮助找到写错的位置。

  • 更重要的是它的万能模块加载理念，即所有的资源都可以且也应该模块化。资源模块化后，有三个好处：

    • 依赖关系单一化。所有CSS和图片等资源的依赖关系统一走JS路线，无需额外处理CSS预处理器的依赖关系，也不需处理代码迁移时的图片合并、字体图片等路径问题；

    • 资源处理集成化。现在可以用loader对各种资源做各种事情，比如复杂的vue-loader等等。

    • 项目结构清晰化。使用Webpack后，你的项目结构总可以表示成这样的函数：dest = webpack(src, config)

工作方式

  1. 初始化配置参数。合并从shell传入和webpack.config.js文件里配置的参数，生产最后的配置结果。

  2. 绑定事件钩子回调。注册所有配置的插件，好让插件监听webpack构建生命周期的事件节点，以做出对应的反应。

  3. 确定Entry逐一遍历。解析文件构建AST语法树，找出每个文件所依赖的文件，递归下去。

  4. 使用loader编译文件。根据文件类型和loader配置找出合适的loader用来对文件进行转换。

  5. 输出文件。递归完成后得到每个文件的最终结果，根据entry配置生成代码块chunk。并输出所有chunk到文件系统。

抽象语法树（abstract syntax code，AST）是源代码的抽象语法结构的树状表示，树上的每个节点都表示源代码中的一种结构，之所以说是抽象的，是因为抽象语法树并不会表示出真实语法出现的每一个细节，比如说，嵌套括号被隐含在树的结构中，并没有以节点的形式呈现。目的是将书写的字符串文件转换成计算机容易识别的数据结构，在计算机上的表现形式就是一个单纯的Object。

运行流程解析

  Webpack是一条生产线，经过一系列处理流程后将源文件传换成输出结果，插件是处理流程中的一个功能。Tapable（Webpack自带的模块，随Webpack一并被安装）组织这条生产线。Webpack运行中广播事件，插件监听它关心的事件。Tapable原理是EventEmit，通过发布者-订阅者模式实现。核心代码：

  Webpack在浏览器实现模块化的本质是将所有代码注入到同一个js文件中，最后生成一个IIFE，引入的所有模块都被一个function包起来组装成一个对象，这个对象作为IIFE的实参被传递进去。

  整个构建过程主要花费事件的部分是递归遍历各个entry寻找依赖逐个编译的过程。每次递归经历String -> AST -> String的流程，经过loader还需要处理一些字符串或执行js脚本。

loader

  真正起编译作用的就是loader，本质是一个普通的function，传入匹配的文件内容，对字符串做处理。

  多个loader存在时，执行顺序是从后往前。

  在loader文件里，可以exports一个pitch的函数，它会先于所有的loader执行。一个rule里的所有的loader在执行时都能拿到这个值。

  常见loader区分：

  • PostCSS：把css解析成JavaScript可以操作的AST，并调用插件处理AST得到结果
  • css-loader：使用@import和url(…)的方式实现require的功能
  • style-loader：将所有计算和的样式加入到页面中，与css-loader组合在一起使样式表嵌入到webpack打包后的js文件中
  • file-loader：解决图片引入路径问题。样式中url路径是相对入口文件html引入的，不是原始css文件所在的路径，file-loader会帮忙做一个转换
  • url-loader：小于limit限制的图片，url-loader会将引入的图片编码生成dataURL一串字符，大于limit限制的图片用file-loader处理。url-loader封装了file-loader。

plugin

Webpack插件起到的作用，就是为事件挂载回调函数，或执行指定脚本。本质是一个带有apply方法的class。

  常用插件及说明：

  • html-webpack-plugin：生成html模板，4版本中chunksSortMode可能会有问题
  • ExtractTextWebpackPlugin–> mini-css-extract-plugin：从js打包文件中抽离css文件
  • UglifyJsParallelPlugin：多线程压缩
  • webpack-sftp-plugin：上传文件到ftp服务器
  • CommonsChunkPlugin–>splitChunks：模块拆分
  • AddAssetHtmlPlugin：配合生成某文件自动加载在html中
